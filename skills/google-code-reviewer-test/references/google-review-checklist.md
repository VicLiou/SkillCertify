# Google Code Review 審查標準清單 (Google Review Checklist)

本文件整理自 Google 官方的代碼審查指南 ([Google's Engineering Practices](https://google.github.io/eng-practices/review/))。在審查程式碼時，必須嚴格遵守以下標準。

---

## 1. 審查核心原則

- **標準 (The Standard of Code Review)**：
  - 代碼審查的目標是**提升代碼庫的整體品質**。
  - 沒有「完美」的程式碼，只有「更好」的程式碼。如果變更使代碼庫整體改善，且沒有引入新的缺陷，原則上應予通過。
  - **不要阻礙進步**：如果修改是合理的，即使還有更好的寫法，也不應編審查者的個人偏好而阻礙合併。

- **區分問題嚴重程度**：
  - **Blockers (重大阻礙)**：涉及功能錯誤、安全漏洞、嚴重效能問題、缺乏測試或設計有重大缺陷。開發者**必須修正**後才能合併。
  - **Warnings (建議改善)**：程式碼可讀性、邊界情況處理或中度設計問題。開發者應儘量修正，但有合理理由時可與審查者討論。
  - **Nits (微小建議)**：關於風格、個人偏好、或細微的最佳化。必須在註解前加上 `Nit: ` 標記，開發者有權拒絕修改，不應阻礙合併。

## 2. 審查檢核項目

### 2.1 設計 (Design)
- 程式碼是否放置在適當的模組中？
- API 設計是否合理、清晰？
- 此變更是否與系統的其餘部分良好整合？
- 是否符合開放封閉原則、單一職責等好的物件導向設計？

### 2.2 功能性 (Functionality)
- 程式碼是否確實達成了開發者的意圖？
- 邏輯是否正確？有無死角（如邊界條件處理、空值檢查、異常處理）？
- 對於使用者或系統的行為，有沒有可能造成意料之外的副作用？

### 2.3 複雜度 (Complexity)
- 程式碼是否過於複雜？
- 每一行、每個函數是否都容易理解？
- 是否存在「過度設計 (YAGNI)」？開發者是否為了「未來可能用到」而寫了目前不需要的抽象化？

### 2.4 測試 (Tests)
- 每個非微小 (non-trivial) 的程式碼變更是否都有對應的單元測試或整合測試？
- 測試是否真的有測到邊界條件與異常流程？
- 測試代碼本身的品質是否與主程式碼相當？測試是否易讀、無冗餘？

### 2.5 命名 (Naming)
- 變數、函數、類別與檔案命名是否具有自解釋性？
- 命名是否能準確描述其用途、代表什麼？（避免使用如 `temp`、`data`、`handler` 等模稜兩可的單字，除非在極短的作用域中）。

### 2.6 註解 (Comments)
- 註解是否解釋了「為什麼 (Why)」要這樣做，而不是描述「做什麼 (What)」？
- 程式碼本身是否足夠清晰到不需要該註解？（如果是，應重構程式碼，刪除冗餘註解）。
- 是否有遺留的 TODO 註解？（如果有，必須標明負責人或 Issue 編號，如 `TODO(username): description`）。

### 2.7 風格與一致性 (Style & Consistency)
- 程式碼是否符合該語言的官方風格指南（例如 Python 的 PEP8、Java 的 Google Style Guide 等）？
- 是否與專案中現有的程式碼風格保持一致？

## 3. 衝突解決機制

1. **優先尊重開發者**：若屬於個人偏好或多種可行方案之一，且沒有客觀的優劣之分，請尊重開發者的選擇。
2. **尋求客觀依據**：若有爭議，以官方文檔、語言規範或設計模式手冊為準，而非個人直覺。
3. **引入第三方意見**：若雙方仍無法達成共識，應升級請教團隊其他成員或 Tech Lead。
